# EdgeRouter X — full edge build

Deploy guide for running cloudflared + pntgw + a Starshield-fed NTP server on a Ubiquiti EdgeRouter X (MT7621A, MIPS LE, 256 MB / 256 MB). EdgeOS v3, Vyatta config, no apt sources, no Python 3 — everything custom has to either fit in the existing userland or be a static Go binary.

## Topology

```
[ISP]
    |
  eth1  WAN (DHCP from Starshield, 192.168.1.x)
    |
[ ER-X ] -- cloudflared --> CF edge --> WARP clients
    |       pntgw          NMEA TCP + UDP
    |       ntpd            stratum-1 via dish
    |
    +-- eth0  192.168.10.1/24  LAN0 (management)
    +-- eth2  192.168.2.1/24   LAN2
    +-- eth3  192.168.3.1/24   LAN3
    +-- eth4  192.168.4.1/24   LAN4
```

Starshield is plugged into eth1 (WAN). The dish NATs to `192.168.1.0/24` on its LAN-side, so the ER-X management LAN must not also be `192.168.1.0/24` — keep them disjoint (this is the most common deployment mistake).

The dish always serves its own gRPC + NTP on `192.168.100.1`, reachable from the ER-X over eth1 regardless of the DHCP subnet.

## 1. Factory bootstrap

After holding the reset pinhole for ~10s on a powered ER-X, only eth0 is active as `192.168.1.1/24` with DHCP. Plug the laptop into eth0, hit `https://192.168.1.1`, change the password.

### Persistent SSH key auth

EdgeOS regenerates `~/.ssh/authorized_keys` from Vyatta config on every commit — keys dropped manually get wiped. Install through the config:

```sh
PUBKEY=$(awk '{print $2}' ~/.ssh/id_ed25519.pub)
ssh ubnt@192.168.1.1 'bash -s' <<REMOTE
. /opt/vyatta/etc/functions/script-template
configure
set system login user ubnt authentication public-keys laptop key '$PUBKEY'
set system login user ubnt authentication public-keys laptop type ssh-ed25519
commit
save
exit
REMOTE
```

### Renumber the management LAN

If the Starshield will plug into eth1, the dish's own 192.168.1.0/24 will collide with the default ER-X LAN. Move eth0 to something disjoint:

```sh
ssh ubnt@192.168.1.1 'bash -s' <<'REMOTE'
. /opt/vyatta/etc/functions/script-template
configure
delete interfaces ethernet eth0 address 192.168.1.1/24
set interfaces ethernet eth0 address 192.168.10.1/24
delete service dhcp-server shared-network-name LAN
set service dhcp-server shared-network-name LAN authoritative enable
set service dhcp-server shared-network-name LAN subnet 192.168.10.0/24 default-router 192.168.10.1
set service dhcp-server shared-network-name LAN subnet 192.168.10.0/24 dns-server 192.168.10.1
set service dhcp-server shared-network-name LAN subnet 192.168.10.0/24 lease 86400
set service dhcp-server shared-network-name LAN subnet 192.168.10.0/24 start 192.168.10.50 stop 192.168.10.150
commit
save
exit
REMOTE
```

This will boot the SSH session. Reconnect by setting your laptop to DHCP on the LAN port — you'll get `192.168.10.x` from the new server.

## 2. Network: WAN + 4 LANs + NAT + DHCP + DNS

eth1 is already configured as DHCP WAN by the basic-setup wizard. Add the rest:

```sh
ssh ubnt@192.168.10.1 'bash -s' <<'REMOTE'
. /opt/vyatta/etc/functions/script-template
configure
# NAT masquerade (LAN -> WAN)
set service nat rule 5010 description 'masquerade lan to wan'
set service nat rule 5010 outbound-interface eth1
set service nat rule 5010 type masquerade
# DNS forwarder (1.1.1.1 upstream, cache enabled)
set service dns forwarding cache-size 300
set service dns forwarding name-server 1.1.1.1
set service dns forwarding name-server 1.0.0.1
set service dns forwarding listen-on eth0
# eth2/3/4 each get their own /24 LAN with DHCP
for i in 2 3 4; do
  set interfaces ethernet eth$i address 192.168.$i.1/24
  set interfaces ethernet eth$i description LAN$i
  set service dhcp-server shared-network-name LAN$i authoritative enable
  set service dhcp-server shared-network-name LAN$i subnet 192.168.$i.0/24 default-router 192.168.$i.1
  set service dhcp-server shared-network-name LAN$i subnet 192.168.$i.0/24 dns-server 192.168.$i.1
  set service dhcp-server shared-network-name LAN$i subnet 192.168.$i.0/24 lease 86400
  set service dhcp-server shared-network-name LAN$i subnet 192.168.$i.0/24 start 192.168.$i.50 stop 192.168.$i.150
  set service dns forwarding listen-on eth$i
done
commit
save
exit
REMOTE
```

Plug ISP/Starshield into eth1. Verify:

```sh
ssh ubnt@192.168.10.1 "ip -4 addr show eth1; ip -4 route; ping -c 2 1.1.1.1; ping -c 2 192.168.100.1"
```

## 3. NTP — Starshield as stratum 1 + fix the rate-limit KoD

```sh
ssh ubnt@192.168.10.1 'bash -s' <<'REMOTE'
. /opt/vyatta/etc/functions/script-template
configure
set system ntp server 192.168.100.1
commit
save
exit
REMOTE
```

EdgeOS hard-codes `limited` in its autogenerated `/etc/ntp.conf` restrict line. With `limited`, ntpd sends RATE kiss-of-death packets to clients that poll quickly, which breaks `sntp` and any actively-syncing app. Strip it on every config commit via a post-config hook:

```sh
ssh ubnt@192.168.10.1 'cat > /config/scripts/post-config.d/ntp-no-rate-limit.sh' <<'HOOK'
#!/bin/sh
NTPCONF=/etc/ntp.conf
if grep -qE '^restrict -[46] default.* limited' "$NTPCONF"; then
  sed -i 's/^\(restrict -[46] default.*\) limited/\1/' "$NTPCONF"
  systemctl restart ntp 2>/dev/null || /etc/init.d/ntp restart 2>/dev/null || true
fi
HOOK
ssh ubnt@192.168.10.1 'chmod +x /config/scripts/post-config.d/ntp-no-rate-limit.sh && sudo /config/scripts/post-config.d/ntp-no-rate-limit.sh'
```

Verify with `ntpq -p`: `*192.168.100.1` with stratum 1 means dish is the active upstream. LAN clients just point at any of the ER-X LAN IPs (`192.168.10.1`, `.2.1`, `.3.1`, `.4.1`) on UDP/123.

## 4. cloudflared — Cloudflare Tunnel on MIPS

Cloudflare doesn't ship MIPS binaries. Build one ourselves from the open source.

### Cross-compile

```sh
git clone --depth 1 https://github.com/cloudflare/cloudflared.git
cd cloudflared
CGO_ENABLED=0 GOOS=linux GOARCH=mipsle go build \
  -mod=vendor -ldflags="-s -w" -o cloudflared-mipsle ./cmd/cloudflared
file cloudflared-mipsle    # ELF 32-bit LSB MIPS32, statically linked, ~27 MB
```

Push to the ER-X:

```sh
scp cloudflared-mipsle ubnt@192.168.10.1:/config/cloudflared
ssh ubnt@192.168.10.1 "chmod +x /config/cloudflared && /config/cloudflared --version"
```

`/config` is on the persistent UBI partition — survives firmware upgrades.

### Tunnel + private network routes via API

Scoped API token with `Account.Cloudflare Tunnel:Edit` + `Zero Trust:Edit`. **Don't** use the Global API Key in scripts.

```sh
ACCT=<account_id>
export CF_TOKEN=<scoped_token>

# 1. Create a remote-managed tunnel
TUN=$(curl -sS -X POST "https://api.cloudflare.com/client/v4/accounts/$ACCT/cfd_tunnel" \
  -H "Authorization: Bearer $CF_TOKEN" -H "Content-Type: application/json" \
  --data '{"name":"er-x-home","config_src":"cloudflare"}' \
  | python3 -c "import sys,json;print(json.load(sys.stdin)['result']['id'])")

# 2. Fetch the connector token, save to file, never paste in chat
curl -sS "https://api.cloudflare.com/client/v4/accounts/$ACCT/cfd_tunnel/$TUN/token" \
  -H "Authorization: Bearer $CF_TOKEN" \
  | python3 -c "import sys,json;print(json.load(sys.stdin)['result'])" \
  > /tmp/.cf_tunnel_token
chmod 600 /tmp/.cf_tunnel_token

# 3. Advertise each LAN as a WARP-routable private network
for NET in 192.168.1.0/24 192.168.2.0/24 192.168.3.0/24 192.168.4.0/24 192.168.10.0/24; do
  curl -sS -X POST "https://api.cloudflare.com/client/v4/accounts/$ACCT/teamnet/routes" \
    -H "Authorization: Bearer $CF_TOKEN" -H "Content-Type: application/json" \
    --data "{\"network\":\"$NET\",\"tunnel_id\":\"$TUN\",\"comment\":\"ER-X LAN\"}"
done
```

### Split-tunnel surgery

The default WARP split-tunnel excludes `192.168.0.0/16`. Replace that single entry with a complement set that excludes everything in that /16 except your ER-X LANs (1, 2, 3, 4, 10):

```sh
EXCLUDE='[
  {"address":"10.0.0.0/8"},{"address":"100.64.0.0/10"},
  {"address":"169.254.0.0/16"},{"address":"172.16.0.0/12"},
  {"address":"192.0.0.0/24"},
  {"address":"192.168.0.0/24"},
  {"address":"192.168.5.0/24"},{"address":"192.168.6.0/23"},
  {"address":"192.168.8.0/23"},{"address":"192.168.11.0/24"},
  {"address":"192.168.12.0/22"},{"address":"192.168.16.0/20"},
  {"address":"192.168.32.0/19"},{"address":"192.168.64.0/18"},
  {"address":"192.168.128.0/17"},
  {"address":"224.0.0.0/24"},{"address":"240.0.0.0/4"},
  {"address":"255.255.255.255/32"},
  {"address":"fe80::/10"},{"address":"fd00::/8"},
  {"address":"ff01::/16"},{"address":"ff02::/16"},
  {"address":"ff03::/16"},{"address":"ff04::/16"},{"address":"ff05::/16"}
]'
curl -sS -X PUT "https://api.cloudflare.com/client/v4/accounts/$ACCT/devices/policy/exclude" \
  -H "Authorization: Bearer $CF_TOKEN" -H "Content-Type: application/json" \
  --data "$EXCLUDE"
```

192.168.0.0/24 (your wifi) and 192.168.8.0/23 (homelab) stay direct; 192.168.{1,2,3,4,10}.0/24 go via WARP through the tunnel.

### Protocol — pin WireGuard

MASQUE/HTTP3 can fail the in-tunnel DNS connectivity check on double-NAT'd networks. WireGuard is the most reliable for this use case:

```sh
curl -sS -X PATCH "https://api.cloudflare.com/client/v4/accounts/$ACCT/devices/policy" \
  -H "Authorization: Bearer $CF_TOKEN" -H "Content-Type: application/json" \
  --data '{"tunnel_protocol":"wireguard","disable_auto_fallback":true}'
```

### Persistent service

`/config` is group-writable to `vyattacfg` (`ubnt` is in that group), so file writes don't need sudo. The systemd unit must live in `/etc/systemd/system/` (root-owned), and EdgeOS's overlay filesystem doesn't guarantee `/etc/` persistence. The fix is to re-install the unit from a post-config hook on every boot — that directory survives firmware upgrades.

Push the token:

```sh
scp /tmp/.cf_tunnel_token ubnt@192.168.10.1:/tmp/cf_tunnel_token
ssh ubnt@192.168.10.1 'sudo mv /tmp/cf_tunnel_token /config/cf_tunnel_token \
  && sudo chown root:root /config/cf_tunnel_token \
  && sudo chmod 600 /config/cf_tunnel_token'
```

Hook:

```sh
ssh ubnt@192.168.10.1 'cat > /config/scripts/post-config.d/cloudflared.sh' <<'HOOK'
#!/bin/sh
# Idempotent boot-time installer for cloudflared.

# Unprivileged ICMP for cloudflared's ping proxy
SYSCTL=/etc/sysctl.d/99-cloudflared-icmp.conf
echo "net.ipv4.ping_group_range = 0 2147483647" > "$SYSCTL"
sysctl -p "$SYSCTL" >/dev/null 2>&1 || true

UNIT=/etc/systemd/system/cloudflared.service
cat > "$UNIT" <<'UNITEOF'
[Unit]
Description=cloudflared tunnel
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
ExecStart=/config/cloudflared --no-autoupdate tunnel run --token-file /config/cf_tunnel_token
Restart=always
RestartSec=10
User=root
LimitNOFILE=65536

[Install]
WantedBy=multi-user.target
UNITEOF

chmod 644 "$UNIT"
systemctl daemon-reload
systemctl enable cloudflared.service >/dev/null 2>&1 || true
systemctl restart cloudflared.service
HOOK
ssh ubnt@192.168.10.1 'chmod +x /config/scripts/post-config.d/cloudflared.sh \
  && sudo /config/scripts/post-config.d/cloudflared.sh'
```

Verify:

```sh
ssh ubnt@192.168.10.1 "systemctl is-active cloudflared"
# active

curl -sS "https://api.cloudflare.com/client/v4/accounts/$ACCT/cfd_tunnel/$TUN" \
  -H "Authorization: Bearer $CF_TOKEN" \
  | python3 -c "import sys,json; d=json.load(sys.stdin)['result']; print(d['status'], len(d['connections']),'conns')"
# healthy 4 conns
```

From a WARP-enrolled client:

```sh
warp-cli connect
ping -c 3 192.168.10.1
ssh ubnt@192.168.10.1 "uptime"
warp-cli disconnect
```

## 5. pntgw — Starshield NMEA gateway

Source in this repo. Build, deploy:

```sh
make erx
scp pntgw-mipsle ubnt@192.168.10.1:/config/pntgw
ssh ubnt@192.168.10.1 "sudo chmod +x /config/pntgw"
```

Persistent service:

```sh
ssh ubnt@192.168.10.1 'cat > /config/scripts/post-config.d/pntgw.sh' <<'HOOK'
#!/bin/sh
UNIT=/etc/systemd/system/pntgw.service
cat > "$UNIT" <<'UNITEOF'
[Unit]
Description=pntgw — Starshield PNT gateway
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
ExecStart=/config/pntgw --config /config/pntgw.yaml
Restart=always
RestartSec=5
User=root
LimitNOFILE=65536

[Install]
WantedBy=multi-user.target
UNITEOF
chmod 644 "$UNIT"
systemctl daemon-reload
systemctl enable pntgw.service >/dev/null 2>&1 || true
systemctl restart pntgw.service
HOOK
ssh ubnt@192.168.10.1 'chmod +x /config/scripts/post-config.d/pntgw.sh \
  && sudo /config/scripts/post-config.d/pntgw.sh'
```

Web UI on `http://192.168.10.1:8080`. NMEA stream on `tcp://192.168.10.1:10110`. Add UDP destinations through the UI.

## 6. Swapping the tunnel to a different Cloudflare account

Only `/config/cf_tunnel_token` is account-specific. Everything else (binary, hook, EdgeOS network config) carries over.

```sh
# In the NEW account
NEW_ACCT=<new_account_id>
export CF_TOKEN=<new_scoped_token>

NEW_TUN=$(curl -sS -X POST "https://api.cloudflare.com/client/v4/accounts/$NEW_ACCT/cfd_tunnel" \
  -H "Authorization: Bearer $CF_TOKEN" -H "Content-Type: application/json" \
  --data '{"name":"er-x-home","config_src":"cloudflare"}' \
  | python3 -c "import sys,json;print(json.load(sys.stdin)['result']['id'])")

curl -sS "https://api.cloudflare.com/client/v4/accounts/$NEW_ACCT/cfd_tunnel/$NEW_TUN/token" \
  -H "Authorization: Bearer $CF_TOKEN" \
  | python3 -c "import sys,json;print(json.load(sys.stdin)['result'])" > /tmp/.cf_tunnel_token
chmod 600 /tmp/.cf_tunnel_token

for NET in 192.168.1.0/24 192.168.2.0/24 192.168.3.0/24 192.168.4.0/24 192.168.10.0/24; do
  curl -sS -X POST "https://api.cloudflare.com/client/v4/accounts/$NEW_ACCT/teamnet/routes" \
    -H "Authorization: Bearer $CF_TOKEN" -H "Content-Type: application/json" \
    --data "{\"network\":\"$NET\",\"tunnel_id\":\"$NEW_TUN\",\"comment\":\"ER-X LAN\"}"
done

# Re-apply split-tunnel + protocol pins on the new account's policy
# (same PUT /devices/policy/exclude and PATCH /devices/policy as section 4)

# Hot-swap the token on the ER-X — no rebuild
scp /tmp/.cf_tunnel_token ubnt@192.168.10.1:/tmp/cf_tunnel_token
ssh ubnt@192.168.10.1 'sudo mv /tmp/cf_tunnel_token /config/cf_tunnel_token \
  && sudo chown root:root /config/cf_tunnel_token \
  && sudo chmod 600 /config/cf_tunnel_token \
  && sudo systemctl restart cloudflared'

# Delete the orphaned tunnel in the OLD account (optional cleanup)
curl -sS -X DELETE "https://api.cloudflare.com/client/v4/accounts/$OLD_ACCT/cfd_tunnel/$OLD_TUN" \
  -H "Authorization: Bearer $OLD_CF_TOKEN"
```

Within ~10s of the restart the new account's tunnel goes `healthy` with 4 connections. WARP clients on the new account get the LAN routes after their next policy refresh.

If the new account uses different LAN CIDRs, change the EdgeOS interface IPs first — `teamnet/routes` rejects overlapping CIDRs across tunnels in the same account.

## Operational notes

| Concern | Location |
|---|---|
| cloudflared binary | `/config/cloudflared` |
| Tunnel token | `/config/cf_tunnel_token` (root:root 600) |
| cloudflared unit | `/etc/systemd/system/cloudflared.service` (re-created on boot) |
| cloudflared boot hook | `/config/scripts/post-config.d/cloudflared.sh` |
| ICMP sysctl | `/etc/sysctl.d/99-cloudflared-icmp.conf` (re-applied by hook) |
| pntgw binary | `/config/pntgw` |
| pntgw config | `/config/pntgw.yaml` |
| pntgw unit | `/etc/systemd/system/pntgw.service` (re-created on boot) |
| pntgw boot hook | `/config/scripts/post-config.d/pntgw.sh` |
| NTP rate-limit hook | `/config/scripts/post-config.d/ntp-no-rate-limit.sh` |
| Persistent journald | not configured — `journalctl -u <svc>` is empty after reboot |

To upgrade either binary: rebuild static MIPS LE, `scp` over the old one, `sudo systemctl restart <svc>`. No package manager.

To rotate the cloudflared tunnel token without recreating the tunnel: `POST /accounts/$ACCT/cfd_tunnel/$TUN/rotate_token`, re-push, restart.

To revoke a WARP-enrolled device: Zero Trust → My Team → Devices → revoke.
